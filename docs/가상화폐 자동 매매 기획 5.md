# 가상화폐 자동 매매 기획 v5.0 — Capital First Rebuild

> 본 문서는 v4.x 전면 폐기 후 **자본 보존 우선(Capital Preservation First)** 원칙으로 다시 쓴 기획서다.
> 작성일: 2026-05-25 · 작성 근거: 실거래 누적 -91.23% 손실(30만원 → 2.6만원)에 대한 사후 분석.

---

## 0. Executive Summary (한 페이지 요약)

### 0.1 v4.x 가 실패한 이유 (실측)

| # | 원인 | 증거 |
|---|------|------|
| 1 | **수수료/슬리피지 누적** | 1분 주기 × Top5 마켓 × 다수 엔진 진입 → 일일 수십 회 주문. 업비트 왕복 0.1% × 50회 = 일 -5% |
| 2 | **상호 충돌하는 8개 엔진** | `engine / enhanced_engine / hybrid_engine / integrated_engine / multi_timeframe_engine / autonomous_orchestrator / personas / emergency_trader` 가 동시에 신호 생성 → 디버깅 불가 |
| 3 | **ML 예측의 통계적 무근거** | 훈련 정확도 85%, 실전 1~33%. 1~5분 단위 가격 방향 예측은 시장 효율성으로 인해 본질적으로 어려운 문제 |
| 4 | **폴링 기반 지연** | 1~3분 주기 REST 폴링. 변동성 구간에서는 신호 시점과 체결 시점의 가격 괴리가 손실 핵심 |
| 5 | **리스크 가드 우회** | min_confidence 0.75 등 임계값은 있었으나, 다중 엔진이 우회 경로를 만듦. 일일 손실 한도가 실제로는 작동하지 않음 |
| 6 | **백테스트 부재** | 모든 파라미터가 실거래에서 튜닝됨 = 실거래 자금이 학습비용 |

### 0.2 v5.0 의 핵심 결정 5가지

1. **단일 엔진(Single Engine)** — 8개 엔진을 폐기하고 `TradingEngine` 한 개만 남긴다.
2. **WebSocket 우선(Stream-First)** — 시세/체결/호가는 업비트 WebSocket(`wss://api.upbit.com/websocket/v1`)으로 수신. REST는 주문/계좌/캔들 부트스트랩에만 사용.
3. **Regime-Switching 전략** — 추세장에서는 Trend-Following, 횡보장에서는 Mean Reversion. 카오스(고변동성/저유동성)에서는 거래 정지. 매 분 재평가.
4. **Risk-First 사이징** — 트레이드당 자본의 1% 손실 = 최대 허용 위험. 포지션 크기는 손절폭에서 역산.
5. **Paper / Live Shadow 병행** — 동일 신호를 Paper 와 Live 양쪽에 동시 발행, **두 PnL 곡선을 비교**하여 슬리피지/수수료 영향을 정량 측정. 실거래 자동매매는 기본 비활성, 명시적으로 켜야 동작.

### 0.3 v5.0 의 수익 목표 — 솔직한 베이스라인

> 광고에서 말하는 "월 20%"는 거짓말이다. 실제 알고리즘 트레이더의 통계적 기대치는 다음과 같다.

| 시나리오 | 월 기대 수익률 | 최대 낙폭(MDD) | 확률 |
|----------|-----------------|----------------|------|
| 최선(strategy edge 존재) | +2 ~ +5% | -8% | ~25% |
| 중간(시장과 동행) | -1 ~ +1% | -10% | ~50% |
| 최악(edge 음수) | -5 ~ -15% | -25% | ~25% |

**v5.0 의 1차 목표는 "최악 시나리오에서도 MDD를 -10% 이내로 묶는 것"이다.** 수익은 그 다음이다.

## 3. 데이터 계층 — WebSocket First

### 3.1 업비트 WebSocket 채널

| 채널 | 용도 | 주기 |
|------|------|------|
| `ticker` | 현재가/24h 거래량/체결강도 | 변동 시 push (~ms) |
| `trade` | 개별 체결 (가격, 수량, ask/bid) | 체결 시 push |
| `orderbook` | 호가 (상위 15호가) | 변동 시 push |

- 연결: `wss://api.upbit.com/websocket/v1`
- 인증 불필요 (public). 주문은 REST.
- 재연결: exponential backoff (1s → 30s cap), ping 120s 간격.

### 3.2 인메모리 저장소 `MarketDataStore`

마켓별로 다음을 보유 (deque, 고정 길이):
- `ticker_latest: Ticker` — 최신 ticker 1개
- `trades: Deque[Trade]` — 최근 2,000 체결
- `orderbook_latest: Orderbook`
- `candles_1m: Deque[Candle]` — 최근 240개 (4시간)
- `candles_5m: Deque[Candle]` — 최근 200개
- `candles_15m: Deque[Candle]` — 최근 200개

### 3.3 캔들 빌더

- 부팅 시 REST `/v1/candles/minutes/{1,5,15}` 로 과거 캔들 N개를 부트스트랩.
- 이후 `trade` 스트림을 받아 현재 진행 중인 분봉을 incremental update.
- 분봉 마감 시점에 deque 추가.

### 3.4 거래 사이클 트리거

- 기본은 **이벤트 드리븐**: 1분봉 마감 이벤트 → `TradingEngine.evaluate(market)` 호출.
- 안전망: 매 30초 polling sweep 으로 stuck 상태 회피.

## 4. 전략 계층

### 4.1 Regime Classifier

추세장(TREND), 횡보장(RANGE), 카오스(CHAOS) 셋 중 하나로 분류. 매 1분봉 마감 시 재평가.

**입력**: 최근 60분 1분봉
**지표**:
- `ADX(14)` — 14 미만이면 횡보, 25 이상이면 추세
- `ATR%(14) = ATR / Close` — 1.5% 초과 시 카오스 의심
- `BB width(20)` — 변동성 압축/팽창
- `Volume Z-score(60)` — |z| > 3 이면 카오스

**판정 규칙** (우선순위):
1. `ATR% > 1.5%` 또는 `|volume_z| > 3` → **CHAOS** (거래 금지)
2. `ADX >= 25` → **TREND**
3. `ADX < 25` → **RANGE** (Mean-Reversion 자체가 RSI<30 & 가격<BB하단을 추가로 요구하므로 분류 단계에서는 BB 폭을 별도로 강제하지 않는다)
4. ADX/지표 계산 실패 → **NEUTRAL** (거래 금지)

> v5.0 초기 버전은 "ADX < 18" 만 RANGE 로 두어 ADX 18~25 구간이 NEUTRAL 사각지대가 되어 **사실상 어떤 전략도 선택되지 않는** 문제가 있었다 (2026-05 운영 중 발견). 위 규칙은 이를 해소한 v5.1 기준이다.

CHAOS / NEUTRAL 에서는 신규 진입 금지, 보유 포지션은 일반 청산 룰을 따른다.

### 4.2 Trend-Following 전략

**진입 조건 (전부 충족 시 BUY)**
- Regime == TREND
- 가격이 `Donchian(20)` 상단 돌파 (close > max(high, 20bars))
- `EMA(20) > EMA(60)` (단기 추세 상승)
- 직전 캔들 volume > `EMA_volume(20) * 1.3`
- 마지막 BUY 로부터 ≥ 15분 경과 (cooldown)

**청산 규칙**
- 손절: 진입가 - `ATR(14) * 2`
- Trailing: 가격이 +1×ATR 도달 시 손절선을 진입가로 이동(본전 락)
- Trailing 2: 가격이 +2×ATR 도달 시 손절선을 (현재가 - 1×ATR) 로 이동
- 시간 청산: 진입 후 240분 경과 시 시장가 청산
- Regime 이 TREND → RANGE/CHAOS 로 변경되면 시장가 청산

### 4.3 Mean Reversion 전략

**진입 조건 (전부 충족 시 BUY)**
- Regime == RANGE
- `RSI(14) < 30` (과매도)
- 가격이 Bollinger(20, 2σ) 하단 미만
- 최근 60분 변동성(σ) > 평소(σ_30d) 의 0.5 배 — 시장이 죽지 않았을 것
- 직전 30분 거래량 > 24h 평균의 0.8 배

**청산 규칙**
- 익절: BB 중앙선(MA20) 도달 시 청산
- 손절: 진입가 - `ATR(14) * 1.5`
- 시간 청산: 진입 후 90분 경과 시 시장가 청산
- Regime 이 RANGE → TREND/CHAOS 로 변경되면 시장가 청산

### 4.4 신호 우선순위 / 충돌

- 두 전략은 Regime 가 상호 배타이므로 동시에 신호를 발생시킬 수 없다.
- 보유 포지션이 있는 마켓은 신규 진입 불가. **마켓당 동시 보유 1개**.

## 5. 리스크 계층

### 5.1 포지션 사이징 (ATR-based)

```
risk_per_trade_krw = equity * 0.01           # 자본의 1%
stop_distance      = atr14 * stop_atr_mult   # 전략별 1.5 or 2.0
qty                = risk_per_trade_krw / stop_distance
notional           = qty * price
```

- `notional > equity * MAX_POSITION_RATIO(0.25)` 면 캡.
- `notional < 6,000 KRW` 면 진입 보류 (업비트 최소).

### 5.2 RiskGuards (모든 BUY 신호는 통과 필수)

| Guard | 차단 조건 |
|-------|-----------|
| `KillSwitchGuard` | 전역 kill_switch == ON |
| `DailyLossGuard` | 당일 PnL <= -DAILY_LOSS_LIMIT(기본 -3%) |
| `MaxDailyTradesGuard` | 당일 진입 횟수 >= MAX_DAILY_TRADES(기본 6) |
| `ConcurrencyGuard` | 오픈 포지션 >= MAX_OPEN_POSITIONS(기본 1) |
| `CooldownGuard` | 직전 손절 후 cooldown(기본 30분) 미경과 |
| `FeeViabilityGuard` | `expected_tp_pct < (fee_rate*2 + slippage_est) * 1.5` |
| `NewsBlackoutGuard` | 업비트 공지에 해당 종목 키워드 → 1시간 차단 |
| `LiquidityGuard` | 24h 거래대금 < **5,000,000,000 KRW (50억원)** 또는 호가 스프레드 > 0.3% |

> 주의: v5.0 초기 구현에는 임계가 `50_000_000_000` (500억원, 100배 오타) 으로 들어가 KRW-ETH 도 상시 BLOCK 되던 버그가 있었다. v5.1 에서 기획서 값으로 정정.

### 5.3 Kill Switch

- `POST /api/risk/kill-switch` (auth 필요) — 모든 신규 진입 즉시 차단, 옵션으로 전 포지션 시장가 청산.
- Celery / FastAPI 양쪽에서 Redis flag 를 읽어 다음 사이클에서 즉시 반영(<= 1초).
- 운영자는 텔레그램 명령어로도 트리거 가능 (향후).

## 6. 브로커 계층

### 6.1 Broker Protocol

```python
class Broker(Protocol):
    name: str   # "paper" | "live"
    def get_equity(self) -> float: ...
    def get_position(self, market) -> Position | None: ...
    def submit_market_buy(self, market, notional_krw) -> Order: ...
    def submit_market_sell(self, market, qty) -> Order: ...
```

### 6.2 PaperBroker

- 인메모리 잔고/포지션.
- 체결가 = 다음 trade tick (스트림이 있을 때) 또는 현재 ticker 매수1/매도1 평균.
- **슬리피지 0.05% 추가**, **수수료 0.05% 차감** — 보수적으로 시뮬레이션.
- 모든 fill 을 `shadow_compare` 테이블에 저장.

### 6.3 LiveBroker

- pyupbit 래퍼, 최소 주문 6,000 KRW 자동 보정.
- 주문 후 0.5s 대기 후 체결 조회(`get_order`)로 실제 체결가 확인 → `actual_fill_price` 저장.
- 실패 시 1회 재시도, 그 후엔 RiskEvent 로 기록.

### 6.4 Shadow Runner

`TradingEngine` 은 신호 1개당 양쪽 브로커에 발주.
- `LIVE_TRADING_ENABLED=false` (기본): LiveBroker 는 no-op, Paper 만 실제 실행.
- `LIVE_TRADING_ENABLED=true`: 둘 다 실행, 두 PnL 곡선을 매 사이클 DB 에 저장.

대시보드에서 "Paper PnL vs Live PnL" 두 라인을 같이 보여줘 운영자가 slippage cost 를 정량 확인.

## 7. 백테스트

### 7.1 입력
- `data/raw/KRW_*.csv` 의 1분봉 (필요 시 5/15분봉 합성)
- 기간: 직전 90일

### 7.2 모델
- Bar-by-bar event loop.
- 진입가 = 다음 봉 시가, 청산가 = 룰에 따른 봉 내 가정(보수적: 손절은 봉 저가, 익절은 봉 고가에서 체결되지 않은 것으로 가정).
- 수수료 0.05% × 2, 슬리피지 0.05% 모델.

### 7.3 메트릭
- 누적 수익률, 연환산 Sharpe, Sortino, MDD, Win-rate, Profit Factor, 평균 보유시간.
- 모든 파라미터 변경 PR 은 backtest report 첨부 필수.

## 12. 운영 절차 (Runbook)

### 12.1 첫 가동 (Day 1 ~ Day 14)
1. `LIVE_TRADING_ENABLED=false`. Paper 모드만 가동.
2. 14일간 Paper PnL, 거래 횟수, 승률, MDD 관찰.
3. 백테스트 결과와 라이브 Paper 결과의 괴리가 ±30% 이내인지 확인.

### 12.2 라이브 전환 기준 (Go-Live Checklist)
- Paper 14일 누적 수익률 ≥ 0%
- Paper MDD ≤ -5%
- 일일 평균 거래 횟수 ≤ 4
- Kill switch 수동 발동 → 1초 내 신규 진입 차단 확인
- 강제 재시작 후에도 오픈 포지션과 일일 카운터가 DB 에서 복구되는지 확인

### 12.3 일상 운영
- 매일 09시 텔레그램 일일 리포트 (선택): 어제 PnL, 거래 횟수, regime 분포, alerts.
- 매주 일요일 backtest 갱신.

### 12.4 비상 시
- 즉시: `curl -X POST /api/risk/kill-switch -d '{"enable":true,"close_positions":true}'`
- 또는: 텔레그램 `/halt` (구현 시)
- 또는: `systemctl stop autotrader-worker autotrader-scheduler`

## 15. 성공의 정의

> **"3개월 운영 후, MDD -10% 이내, 일일 거래 횟수 평균 ≤ 4, 누적 PnL > 0."**
>

손실 -91%의 진짜 문제는 "큰 수익을 못 낸 것"이 아니라 "원금이 사라진 것"이다. v5.0 은 그것을 막는 시스템이다.
