# 가상화폐 자동 매매 기획 v7.0 — Dynamic Portfolio

> 본 문서는 v5.0/v6.0(고정 2종목·단일 포지션, 자본 보존) 위에 **동적 포트폴리오(Dynamic Universe)**
> 를 얹어 "거래가 거의 일어나지 않던" 문제를 구조적으로 해결한 개정판이다.
> 작성일: 2026-06-02 · 배경: 운영 중단(매매·잔고 조회 정지) 사후 진단 + 유망 코인 자동 선별 요구.

---

## 0. 운영 중단 진단 (Root Cause)

대시보드(`/dashboard`)가 오래 멈춘 현상은 세 층위로 분리된다.

| 층위 | 증상 | 진단 결과 |
|------|------|-----------|
| **전략/코드** | 매매가 거의 0건 | 고정 2종목(BTC/ETH) + v6 초보수 파라미터(단일 포지션, ADX≥25 추세, RSI<30, ATR%>1.5% 카오스 정지)로 **진입 조건이 사실상 동시에 성립하지 않음.** BTC/ETH만으로는 "유망 구간"을 만날 확률이 낮다. |
| **API/거래소** | 잔고 조회 실패 | 업비트 Open API는 **허용 IP 화이트리스트**를 강제한다. 서버(`158.180.71.84`)의 아웃바운드 IP가 키에 등록돼 있지 않으면 `get_balances()`가 `{'error': ...}`(`no_authorization_ip`)를 반환한다. 코드 버그가 아니라 **운영 설정 문제**. |
| **서버/프로세스** | 전반적 정지 | WebSocket 피드가 끊기면 모든 마켓이 stale(>120s) 처리되어 엔진이 스킵 → 외형상 "정지". 동적 유니버스/재구독으로 자가 복구력을 강화. |

→ 코드 개정으로 **(1) 전략 층(동적 포트폴리오)** 과 **(3) 자가복구**를 해결하고,
**(2) API/IP 문제**는 진단 엔드포인트(`/api/account/diagnostics`)로 즉시 원인 파악이 가능하도록 했다.
(2)의 최종 조치는 코드가 아니라 **업비트 콘솔에서 서버 IP를 허용 목록에 추가**하는 운영 작업이다.

## 1. 왜 동적 포트폴리오인가

주식과 달리 암호화폐는 **주도 종목이 빠르게 순환(rotation)**한다. 특정 2종목에 고정하면
대부분의 기회를 놓치고, 그 2종목이 횡보하면 시스템 전체가 "거래 0"에 빠진다.

따라서 v7.0은 **고정 포트폴리오를 폐기**하고, 매 주기마다 전체 KRW 마켓을 스캔해
**유망 코인만 동적으로 선별(Cross-sectional Momentum + Liquidity)** 한다.

### 1.1 투자 기법 — Cross-sectional Momentum + Regime-adaptive Entry

학술·실무에서 가장 견고하게 검증된 체계적 기법을 결합한다.

1. **유니버스 선별 (Cross-sectional Momentum)**
   - 전체 KRW 마켓 → **유동성 필터**(24h 거래대금 ≥ 300억) → **모멘텀 랭킹**(24h 수익률) → 상위 N개.
   - **과열 제외**: 24h +30% 초과 종목은 반전 위험으로 제외(고점 추격 방지).
   - **스테이블코인 제외**(USDT/USDC/DAI 등).
   - **앵커 고정**: BTC/ETH는 항상 포함(시장 베타 유지).
2. **레짐 적응형 진입 (기존 v5 엔진 재사용)**
   - 선별된 각 코인에 대해 TREND→Donchian 돌파+EMA, RANGE→RSI/볼린저 평균회귀, CHAOS→정지.
3. **리스크 우선 사이징**
   - 트레이드당 자본 1% 위험, 단일 포지션 ≤ 25%, **포트폴리오 총노출 ≤ 90%**.

이 조합은 "유망한 곳에서만, 추세/횡보에 맞는 방식으로, 위험을 통제하며" 진입한다.

## 2. 아키텍처 변경

```
UniverseRunner (15분 주기)
  └─ UniverseSelector.select()           # 전체 KRW 마켓 스캔·랭킹
        └─ rank_candidates()             # 순수 함수(단위 테스트 대상)
  └─ ActiveMarketRegistry.set([...])     # 활성 마켓 발행(version++)
        ├─ Upbit WebSocket  ── version 변경 감지 → 재구독
        └─ CandleBuilder.bootstrap_markets(new)  # 신규 마켓 캔들 워밍업

TradingEngine.evaluate_all()
  └─ for market in get_active_markets():  # 고정 리스트가 아닌 동적 레지스트리
        evaluate_market(market)           # 레짐→전략→리스크가드→사이징→주문
        (포트폴리오 총노출 캡 적용)
```

### 2.1 신규/변경 모듈

| 파일 | 역할 |
|------|------|
| `app/strategy/universe.py` (신규) | `UniverseSelector` + 순수 랭킹 함수 `rank_candidates` |
| `app/marketdata/active.py` (신규) | `ActiveMarketRegistry` — 활성 마켓 공유 상태(+버전) |
| `app/engine/universe_runner.py` (신규) | 주기적 유니버스 갱신·신규 마켓 부트스트랩 |
| `app/marketdata/upbit_ws.py` | 레지스트리 기반 **동적 재구독** 지원 |
| `app/marketdata/candles.py` | `bootstrap_markets()` 추가(부분 워밍업) |
| `app/engine/trading_engine.py` | `get_active_markets()` 순회 + **포트폴리오 노출 캡** |
| `app/api/routes/account.py` | `/diagnostics` — 키/IP/네트워크 원인 즉시 식별 |
| `app/api/routes/strategy.py`, `dashboard.py` | 동적 유니버스 노출 |

## 3. 설정 (`.env`)

```env
# 동적 포트폴리오
DYNAMIC_UNIVERSE_ENABLED=true
UNIVERSE_SIZE=6                       # 동시 추적 유망 코인 수
UNIVERSE_REFRESH_SEC=900              # 재선정 주기(15분)
UNIVERSE_MIN_VALUE_24H=30000000000    # 유동성 하한(24h 거래대금 300억)
UNIVERSE_ALWAYS_INCLUDE=KRW-BTC,KRW-ETH

# 포트폴리오 리스크
MAX_OPEN_POSITIONS=4
MAX_POSITION_RATIO=0.25
MAX_PORTFOLIO_EXPOSURE=0.90
RISK_PER_TRADE=0.01
DAILY_LOSS_LIMIT=0.03
MAX_DAILY_TRADES=12
```

## 4. v6 → v7 비교

| 항목 | v6.0 | **v7.0** |
|---|---|---|
| 종목 선택 | 고정 KRW-BTC/ETH | **전체 KRW 동적 선별(유동성+모멘텀)** |
| 동시 포지션 | 1개 | **최대 4개(포트폴리오)** |
| 노출 통제 | 단일 25% | 단일 25% + **총 노출 90% 캡** |
| 일일 거래 | 6회 | 12회 |
| 피드 | 고정 구독 | **유니버스 변경 시 자동 재구독** |
| 진단 | 로그 의존 | **`/api/account/diagnostics`** |

## 6. 운영 체크리스트 (잔고/매매 복구)

1. `.env`에 `UPBIT_ACCESS_KEY/SECRET_KEY` 설정 확인.
2. `GET /api/account/diagnostics` 호출 → `outbound_ip` 확인.
3. 업비트 Open API 콘솔에서 해당 IP를 **허용 IP**에 등록(잔고 조회 실패의 주원인).
4. `balance_ok=true` 확인 후 `LIVE_TRADING_ENABLED=true`로 실매매 활성.
5. 대시보드 `tracked_markets`가 동적 유니버스로 갱신되는지 확인.
